# 媒体別特性ガイド

> 各媒体の特性・読者層・成功パターンをまとめたガイド。
> `/strategy` コマンドで媒体別戦略を策定する際に参照する。

---

## note

### 媒体特性

| 項目 | 内容 |
|------|------|
| 種別 | ブログ・エッセイプラットフォーム |
| 主な流入 | note内レコメンド、SNS（Twitter/X）シェア |
| 記事寿命 | 短〜中期（公開直後〜数週間がピーク） |
| 収益化 | 有料記事、サポート機能 |

### 読者層

- クリエイター・ビジネスパーソン・エンジニア（非技術寄り多め）
- 技術の「なぜ」「どう思うか」に関心がある層
- ストーリー性のある記事を好む
- 通勤・休憩中にスマホで読むことが多い

### 成功パターン

- 体験談・失敗談を交えたストーリー構成
- 導入で共感を呼ぶ問いかけ
- 結論を先に提示し、背景を後から補足
- 1記事1テーマで深堀りするより、広く浅くカバー
- 2,000〜4,000文字が読了率が高い
- 見出し画像（アイキャッチ）の質がクリック率に直結

### 避けるべきパターン

- コードブロックが多すぎる記事（読者層にミスマッチ）
- 前提知識を多く要求する内容
- 淡々とした技術解説のみ

## Qiita

### 媒体特性

| 項目 | 内容 |
|------|------|
| 種別 | 技術情報共有プラットフォーム |
| 主な流入 | Google検索、Qiita内トレンド、タグフォロー |
| 記事寿命 | 長期（検索流入で数年読まれる） |
| 評価指標 | LGTM数、ストック数 |

### 読者層

- 現役エンジニア（ジュニア〜シニア）
- 具体的な技術課題の解決策を探している
- 「動くコード」を求めている
- 業務中にPC/タブレットで参照することが多い

### 成功パターン

- 再現可能なコードサンプル付き
- 問題→原因→解決の明確な流れ
- 環境情報（OS、バージョン等）を明記
- 適切なタグ選定（人気タグ + ニッチタグの組み合わせ）
- 3,000〜6,000文字が最適
- 見出しに技術キーワードを含める

### 避けるべきパターン

- ポエム的・エッセイ的な記述が多い
- コードなしで概念だけ説明
- 古いバージョンの情報を更新せず放置

## Zenn

### 媒体特性

| 項目 | 内容 |
|------|------|
| 種別 | 技術情報共有プラットフォーム（GitHub連携） |
| 主な流入 | Google検索、Zennトレンド、Twitter/X |
| 記事寿命 | 長期（Qiita同様、検索流入が主） |
| 特徴 | Book機能（連載）、Scrap機能（メモ） |
| 評価指標 | いいね数 |

### 読者層

- 現役エンジニア（中級〜上級が多い）
- 新しい技術・ツールへの関心が高い
- 深い技術解説を好む
- Qiitaと重複するが、よりモダン技術志向

### 成功パターン

- 深い技術解説 + 実践的なコード
- 最新技術・ツールの導入記事
- Book機能を活かした連載形式（体系的な学習コンテンツ）
- 3,000〜8,000文字（深い解説は長めでもOK）
- Tech / Idea の記事タイプを適切に選択

### 避けるべきパターン

- 初心者向けすぎる基礎解説（読者層にミスマッチ）
- 特定ベンダーの宣伝色が強い記事

## WordPress（自社ブログ）

### 媒体特性

| 項目 | 内容 |
|------|------|
| 種別 | 自社運営ブログ |
| 主な流入 | Google検索（SEOが主戦場） |
| 記事寿命 | 長期（SEO資産として蓄積） |
| テーマ | engineer-blog v1.0.0（独自テーマ） |
| 特徴 | 自由なデザイン・CTA・内部リンク設計が可能 |

### 読者層

- Google検索で技術課題を調べているエンジニア
- 特定のキーワードで検索してたどり着く
- 記事の信頼性・網羅性を重視
- PC閲覧が多い（業務中の参照）

### 成功パターン

- SEOキーワードを意識したタイトル・見出し構成
- 検索意図（Know/Do/Go/Buy）に沿った記事構造
- 導入→本論→まとめ→CTAの定型構成
- 内部リンクで関連記事を相互接続
- メタディスクリプション・OGPの最適化
- 3,000〜8,000文字（SEO観点では長めが有利）
- Mermaid図解・コードハイライトで視覚的にリッチに

### 避けるべきパターン

- SEOキーワードの過剰挿入（キーワードスタッフィング）
- 薄いコンテンツ（500文字以下）
- 他媒体からのコピー（重複コンテンツペナルティ）

## 社内Git

### 媒体特性

| 項目 | 内容 |
|------|------|
| 種別 | 社内ナレッジリポジトリ（classlab-inc/document/Knowledge） |
| 主な流入 | 社内チャット・Slackでの共有 |
| 記事寿命 | 長期（社内アーカイブとして蓄積） |
| 特徴 | 社内固有の文脈・プロジェクト名を直接使える |

### 読者層

- 社内エンジニア（全レベル）
- プロジェクトの文脈を共有している
- 業務に直結する実践的な情報を求めている
- 社内ツール・プロセスへの適用方法に関心がある

### 成功パターン

- 社内プロジェクト名・ツール名を具体的に記載
- 「社内でどう使うか」の適用セクションを追加
- 社内の既存ドキュメントへのリンク
- 外部記事の要約 + 社内への示唆という構成
- 文字数制限なし（必要な分だけ書く）

### 避けるべきパターン

- 外部記事のコピペ（社内向けの付加価値がない）
- 社内コンテキストなしの一般論のみ
